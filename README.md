# NL → SQL

Natural-language interface to a SQL database. Ask a question in plain English and
receive validated, executable SQL with results and a visualization — backed by
retrieval-augmented prompting, dual-LLM fallback, and query-safety enforcement.

## Overview

`NL → SQL` converts natural-language questions into safe, executable SQL over a
clinic database. Each request is grounded with retrieved few-shot examples,
generated by an LLM with automatic provider fallback, validated against a strict
safety policy, executed, and returned as structured JSON with a Plotly chart.

**Accuracy:** 18/20 evaluation questions (90%), with both failures documented.

---

## Architecture

```
                    ┌─────────────────────────┐
   Question  ─────▶ │  RAG retrieval           │  cosine similarity over
                    │  (15 verified Q→SQL pairs)│  TF-IDF vectors
                    └───────────┬─────────────┘
                                │  top-k examples as few-shot context
                    ┌───────────▼─────────────┐
                    │  LLM generation          │  Groq llama-3.3-70b (primary)
                    │  Groq → Gemini fallback  │  Gemini 2.0 Flash (on failure)
                    └───────────┬─────────────┘
                                │  raw SQL
                    ┌───────────▼─────────────┐
                    │  Safety validator        │  SELECT-only, block writes
                    │                          │  & system tables → HTTP 400
                    └───────────┬─────────────┘
                                │  validated SQL
                    ┌───────────▼─────────────┐
                    │  SQLite execution        │  + 5-min cache, rate limiting
                    └───────────┬─────────────┘
                                │
                    JSON: { sql, rows, model_used, chart, examples_used }
```

---

## Key features

- **Retrieval-augmented generation** — retrieves the most similar verified
  question→SQL pairs (TF-IDF + cosine similarity) and injects them as few-shot
  context, improving accuracy on unseen phrasings.
- **Dual-LLM fallback** — Groq `llama-3.3-70b` primary; automatic failover to
  Gemini 2.0 Flash on any error, so a single provider issue doesn't break the API.
- **SQL safety enforcement** — every generated query must be a single `SELECT`;
  write operations and system-table access are rejected before execution.
- **Production-style API hygiene** — Pydantic validation, MD5-keyed 5-minute
  response cache, and per-IP rate limiting (20 req/min).
- **Auto visualization** — results render as a Plotly bar or line chart based on
  column types.

| Layer | Technology |
|---|---|
| API framework | FastAPI |

## Getting started

### Prerequisites
- Python 3.10+
- A Groq API key ([console.groq.com](https://console.groq.com)) and a Gemini API
  key ([aistudio.google.com/apikey](https://aistudio.google.com/apikey)) — both free.

### Installation
```bash
git clone https://github.com/whizey/NL--SQL.git
cd NL--SQL

python -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate

pip install -r requirements.txt
```

### Configuration
Create a `.env` file in the project root:
```env
GROQ_API_KEY=your_groq_key
GOOGLE_API_KEY=your_gemini_key
```

### Run
```bash
python setup_database.py          # builds clinic.db
uvicorn main:app --port 8000 --reload
```
The app is available at `http://localhost:8000`.

## API

### `POST /chat`
Convert a question to SQL, execute it, and return results.

**Request**
```json
{ "question": "Show me the top 5 patients by total spending" }
```

**Response (200)**
```json
{
  "sql_query": "SELECT p.first_name, p.last_name, SUM(i.total_amount) AS total_spent ...",
  "model_used": "Groq · llama-3.3-70b",
  "columns": ["first_name", "last_name", "total_spent"],
  "rows": [["Aarav", "Sharma", 7820.50], ...],
  "row_count": 5,
  "examples_used": 2,
  "chart": { "...plotly payload..." },
  "cached": false
}
```

**Error codes**

| Status | Meaning |
|---|---|
| 400 | Unsafe SQL (non-SELECT or blocked keyword) |
| 422 | Database execution error |
| 429 | Rate limit exceeded (20 req/min) |
| 503 | Both LLM providers failed |

### `GET /health`
Returns API status, database connectivity, and the number of retrieval examples loaded.

## Database schema

```
patients     (id, first_name, last_name, email, phone, date_of_birth, gender, city, registered_date)   — 200
doctors      (id, name, specialization, department, phone)                                              —  15
appointments (id, patient_id, doctor_id, appointment_date, status, notes)                               — 500
treatments   (id, appointment_id, treatment_name, cost, duration_minutes)                               — 350
invoices     (id, patient_id, doctor_id, invoice_date, total_amount, paid_amount, status)               — 300
```
`invoices` carries `doctor_id` directly, so revenue-by-doctor queries avoid an
unnecessary join through `appointments`. **Total: 1,365 rows.**

## Evaluation

The system was evaluated on 20 questions spanning joins, aggregations, `HAVING`
filters, and date grouping. The run logged the generated SQL and pass/fail per
question.

**Result: 18/20 (90%).** Documented failures:

| # | Question type | Failure |
|---|---|---|
| 13 | Patient-level aggregate | Omitted the join to `patients`, returned IDs instead of names |
| 15 | Date formatting | Returned the numeric weekday instead of converting to a day name |

Reporting the two failures rather than a perfect score is intentional — the
evaluation exists to surface exactly these gaps.
